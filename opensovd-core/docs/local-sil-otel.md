# Local SIL OpenTelemetry spike for `sovd-main`

This Phase 6 prep slice wires one local SIL binary, `sovd-main`, to emit OTLP
request spans when `[logging.otel]` is enabled in its TOML configuration.

## What this slice enables

- `sovd-main` keeps plain terminal logging by default.
- When `[logging.otel].enabled = true`, `sovd-main` exports HTTP request spans
  over OTLP/HTTP to the configured collector endpoint.
- Request spans come from `tower_http::trace::TraceLayer`, so a single
  `GET /sovd/v1/components` request is enough to produce a trace.

## Config snippet

Add or override the following in the config file passed to `sovd-main`:

```toml
[server]
address = "127.0.0.1"
port = 20002

[logging.otel]
enabled = true
endpoint = "http://127.0.0.1:4317"
service_name = "sovd-main"
```

If `endpoint` and `service_name` are omitted, the defaults are:

- `endpoint = "http://127.0.0.1:4317"`
- `service_name = "sovd-main"`

## Local Jaeger verification steps

1. Start a local Jaeger all-in-one collector:

```powershell
docker run -d --name jaeger-verify -p 4317:4317 -p 4318:4318 -p 16686:16686 jaegertracing/all-in-one:latest
```

2. Ensure Jaeger exposes OTLP/gRPC on `127.0.0.1:4317` and the UI on
   `127.0.0.1:16686`.
3. Start `sovd-main` with the config snippet above:

```powershell
cargo run --locked -p sovd-main -- --config-file opensovd.toml
```

4. In another shell, trigger one traced request:

```powershell
curl.exe http://127.0.0.1:20002/sovd/v1/components
```

5. Open the Jaeger UI and confirm the `sovd-main` request span appears:

```powershell
Start-Process "http://127.0.0.1:16686"
```

6. Verify the trace through the Jaeger API as well:

```powershell
curl.exe "http://127.0.0.1:16686/api/traces?service=sovd-main&limit=20"
```

Expected result: the JSON response contains at least one trace for the
`sovd-main` service, and the trace includes an HTTP server span generated by
the request above.

7. Remove the temporary Jaeger container after verification:

```powershell
docker rm -f jaeger-verify
```
